# Moving Blocks Compression Overview

This note covers the first-pass understanding of Acorn Replay format 7,
`Moving Blocks`, from:

- `ARMovie_2003/Video/Decomp7/Docs/Stream`
- `ARMovie_2003/Video/Decomp7/bas/BatchComp,ffb.txt`
- `ARMovie_2003/Video/Decomp7/bas/MakeDecomp,ffb.txt`

Later notes should document the exact bitstream tables and compare format 7
with `Decomp17`, `Decomp19`, and `Decomp20`.

## Format

Moving Blocks is Acorn Replay compression type 7.

Its `Info` file says:

```text
Moving Blocks
Acorn Computers 1993
<variable bpp>
x multiple 4
y multiple 4
Temporal,Spatial
YUV 5,5,5
```

The source frame is treated as 5-bit YUV pixels. The encoder scans the frame in
4x4 blocks from top left to bottom right.

## Basic Coding Model

The stream is a least-significant-bit-first bitstream.

Each 4x4 block can be represented as:

- raw 4x4 data;
- a copy from the previous frame;
- a copy from already-decoded pixels in the current frame;
- four 2x2 sub-block decisions.

Raw 4x4 data is:

```text
16 Y values + 1 U + 1 V
```

Each component is 5 bits, so a raw 4x4 block costs 90 bits. The U and V values
are shared across the whole 4x4 block.

Raw 2x2 data is:

```text
4 Y values + 1 U + 1 V
```

Again components are 5 bits, so a raw 2x2 sub-block costs 30 bits.

## Temporal and Spatial Copies

Temporal copies read from the previous decoded frame. Spatial copies read from
already-decoded areas of the current output frame.

The search radius in `Decomp7` is:

```basic
move%=4
xr%=2*move%+1
```

So temporal matching tests nearby positions up to 4 pixels away in x/y, bounded
at the image edges.

Spatial 4x4 copies are limited to positions that are already available while
scanning:

- two, one, zero, one, or two pixels horizontally while copying from four rows
  above;
- or four pixels left from the same row or earlier rows.

Spatial 2x2 copies use a similar smaller set of offsets.

## Frame Loop

The high-level encoder loop is:

```basic
FOR frame%=iframe TO maxframe%-1 STEP fsr
  PROCExpandNextSourceFrame(frame%)

  calculate uplim% and downlim%
  PROCmatch

  totq% += qual%
  SWAP b1%, b3%
  PROCCheckChunkFinished(b1%)
NEXT
```

The important buffers are:

- `b1%`: previous/current reconstructed frame used as temporal reference;
- `b2%`: source frame expanded by CompLib;
- `b3%`: reconstructed output frame being generated by this encode pass;
- `b4%`: temporary/debug buffer.

The encoder does not just write a stream. It also reconstructs its own decoded
frame into `b3%`. This keeps the compressor's future temporal references in
sync with what the Replay decompressor will reconstruct.

## Per-Block Decision Process

`PROCmatch` scans 4x4 blocks. For each block:

1. Try to find a 4x4 match with `findblk16%`.
2. If a 4x4 match succeeds, emit the temporal/spatial copy code and copy the
   matched pixels into `b3%`.
3. If no 4x4 match succeeds, try four 2x2 decisions with `find4%`.
4. If the four 2x2 decisions take more than 90 bits, abandon them and emit raw
   4x4 data with `sample16%`.

That last rule is important: raw 4x4 data costs 90 bits, so 2x2 subdivision is
only useful if it beats raw coding.

The encoder stores per-block bits with `storebits%`, using `!boploc` as a bit
pointer. After the frame, byte `bop%` is recovered as:

```basic
bop% = (!boploc + 7) >> 3
```

## Matching Criteria

Matching is controlled by four quality parameters loaded from `QP%`:

```basic
!maxi% = QP%(qual%,0)
!maxe% = QP%(qual%,1)
!tot16% = QP%(qual%,2)
!tot4% = QP%(qual%,3)
```

The comments describe these as:

```text
max individual error
max exceptional error
maximum exceptional pixels
total allowable error
```

In practice, the match routines compare absolute component errors against:

- `maxi`: ordinary per-component tolerance;
- `maxe`: exceptional per-component tolerance;
- `tot16` / `tot4`: total allowed accumulated error for 4x4 or 2x2 matching.

The quality table rises with `qual%`. For example:

```text
qual 0:  maxi=0, maxe=0,  tot16=0,   tot4=0
qual 7:  maxi=1, maxe=2,  tot16=14,  tot4=4
qual 15: maxi=1, maxe=3,  tot16=48,  tot4=12
qual 23: maxi=4, maxe=8,  tot16=144, tot4=36
qual 28: maxi=6, maxe=13, tot16=216, tot4=60
```

So larger `qual%` means looser matching and usually smaller output. The name is
slightly misleading if read as "quality level"; in this encoder, raising
`qual%` generally lowers fidelity to hit the byte budget.

## Bitrate and Quality Control

There are three modes.

### `-quality <n>`

When `qd%=1`, the encoder uses the requested `qual%` and effectively disables
the byte target:

```basic
downlim% = sz%
uplim% = 0
```

The initial frame size estimate is:

```basic
framesize = sz% / 16 * 6
```

That is six bytes per 4x4 source block, but in quality mode the actual frame
size is allowed to vary widely.

### `-dirty`

When `qd%=-1`, the encoder sets a simple range around `framesize`:

```basic
downlim% = framesize * 1.025
uplim%   = framesize * 0.90
```

This gives a broad target window and avoids the normal per-chunk correction.

### Default data-rate mode

With neither `-quality` nor `-dirty`, the target byte budget comes from:

```basic
data = (chunktime * double - latency) / double * datarate * 1024 - soundbytes%
framesize = data / fpf
```

Then each frame gets a window around the current target:

```basic
extrafree = chunker% * framesize - bop%
downlim% = framesize + extrafree
if downlim% < framesize * 0.90 then downlim% = framesize * 0.90
```

For lower frame rates:

```basic
uplim%   = downlim% - 500
downlim% = downlim% + 500
```

For 25 fps and above:

```basic
uplim%   = downlim% - 250
downlim% = downlim% + 250
```

This means the encoder spends or recovers budget across a chunk. If previous
frames were small, `extrafree` lets the next frame be larger. If previous
frames were large, the next frame is pushed smaller, but not below 90% of the
nominal `framesize`.

`-arm2` can clamp `framesize`/`downlim%` to an ARM2 playback limit.

## Retry Loop

`PROCmatch` encodes the same frame repeatedly until the size lands inside the
target window or the quality parameter reaches a limit.

Pseudo-code:

```text
start at current qual%
repeat:
    reset output pointer to start of this frame
    encode whole frame with current thresholds
    frame_bytes = new_bop - old_bop

    if frame_bytes < uplim:
        qual% -= 1      ; stricter match, larger output, better fidelity
    else if frame_bytes > downlim:
        qual% += 1      ; looser match, smaller output, worse fidelity
    else:
        accept frame
until accepted
```

The direction is remembered. Once the encoder starts increasing or decreasing
`qual%`, it keeps moving in that direction until the frame is acceptable.

If `qual%` has risen three levels from the starting value for the frame, the
code has a "Cut" path that increases `downlim2%`, allowing a larger-than-normal
frame rather than endlessly chasing a hard target. This is a pragmatic escape
for difficult frames.

## First Frame

The first frame has no temporal reference:

```basic
!fno = frame% - sframe%
```

If `!fno` is zero, the match routines skip previous-frame search. Spatial
matches and raw blocks may still be used.

## Output Statistics

The compressor logs:

- frame number;
- accepted `qual%`;
- encoded byte count;
- encode time;
- `uplim%` and `downlim%`;
- per-chunk average quality;
- count of 4x4 and 2x2 moved blocks.

These statistics should be useful later when validating a C implementation.

## Current Mental Model

Moving Blocks is a block matcher with reconstruction feedback.

It tries cheap copy references first:

1. temporal 4x4 copy from previous frame;
2. spatial 4x4 copy from current frame;
3. four independent 2x2 copy/raw decisions;
4. raw 4x4 block.

The bitrate controller does not directly choose modes per block. Instead, it
adjusts the match tolerances for the whole frame, re-encodes the frame, and
accepts the first version whose byte count fits the current budget window.

Higher `qual%` means more blocks are allowed to match approximately, which
reduces output size but increases distortion. Lower `qual%` forces closer or
exact matches and therefore emits more raw data.
