# Moving Lines Bitstream First Pass

This note documents the first-pass Moving Lines code-word format from:

- `ARMovie_2003/Video/MovingLine/bas/BatchComp,ffb.txt`
- `ARMovie_2003/Video/MovingLine/bas/MakeDecomp,ffb.txt`

It focuses on the code-word families and frame terminator. Decompressor-side
offset-table construction and output-size variants are documented in
`notes/moving-lines-decompressor.md`.

## Stream Unit

Moving Lines is a halfword-oriented stream. The source pointer passed to the
decompressor must be halfword aligned.

The decompressor reads a 16-bit word and derives:

```text
literal_flag = word & 1
payload      = (word >> 1) & 0x7fff
```

If `literal_flag` is zero, the word is a single literal 15-bit pixel.

If `literal_flag` is one, the word is a command. Most commands then split the
15-bit payload as:

```text
direction_or_family = payload >> 6
length_field        = payload & 0x3f
```

The compressor comments summarise the main forms as:

```basic
REM <b5><g5><r5>0       literal pixel
REM <q9><l6>1           copy dirn q, length+2, range 0-&11f
REM <q9><l6>1           copy dirn q, length+2, range &120-&1cb
REM <&1cc><000000>1     end of frame
REM <&1cc><6(<>0)>1     3-65 repetitions of next pixel
REM <&1e><n10>1         skip n+1 pixels
REM <&1f><n10>1 <p15>   literal n pixels (n>15)
```

The "skip" wording is misleading when read as "leave output unchanged". The
decompressor implements it as a same-position copy from the previous decoded
frame.

## Literal Pixel

```text
bit 0      = 0
bits 1..15 = pixel
```

The compressor writes short literal runs as one halfword per pixel:

```basic
PROCopw(b2%!J%<<1)
```

The pixel is a 15-bit source value. `Resources/Info` advertises both
`YUV 5,5,5` and `RGB 5,5,5`; the compressor/decompressor use the same 15-bit
code-word layout and the display tables decide how to present the value.

## Temporal And Spatial Copy

For copy commands:

```text
bit 0       = 1
bits 1..6   = length - 2
bits 7..15  = direction/table code
```

So:

```text
length = ((word >> 1) & 63) + 2
code   = word >> 7
```

The decompressor divides the code range:

| Code range | Meaning |
| --- | --- |
| `0x000..0x11f` | copy from previous decoded frame |
| `0x120..0x1ca` | copy from current decoded frame, as generated by the compressor/init table |

The decompressor dispatch treats any command below `0x1cc` and at or above
`0x120` as spatial. The source comments state:

```basic
REM 000-11f are copy from previous frame
REM 120-1cb are copy from current frame
```

At this pass, `0x1cb` looks reserved or unused: `smove%=9` generates 171
spatial offsets, which covers `0x120..0x1ca`.

The table pointed to by `r5` contains both offset families:

```basic
r5 - temporal table of line offsets (0-&11f) plus spatial offsets (&120-&1cb)
```

For temporal copies, the decompressor adds the inter-frame buffer offset. For
spatial copies, it uses the current output buffer directly.

The compressor builds temporal offsets with `move%=8`, giving a 17x17 search
area. It skips the centre entry in the emitted table code:

```basic
IFQ%>centre% q%(Q%)=((Q%-1)<<7)+1 ELSE q%(Q%)=(Q%<<7)+1
```

That leaves the same-position previous-frame copy to the run command described
below.

The compressor builds spatial offsets with `smove%=9`, but only for rows above
the current row:

```basic
FORJ%=-smove% TO -1
 FORI%=-smove% TO smove%
```

Those entries are emitted as codes starting at `0x120`.

## Same-Position Previous-Frame Run

The compressor accumulates runs where the best match is the previous frame at
the same pixel position:

```basic
IFQ%=centre% THEN
  CALLp1copy%,j%(Q%):skiplen%+=M%
```

It emits those runs in `PROCr`:

```basic
PROCopw(1+(skiplen%-1<<1)+(&1e<<11))
```

This command uses a different split:

```text
bit 0       = 1
bits 1..10  = length - 1
bits 11..15 = 0x1e
```

So:

```text
length = ((word >> 1) & 1023) + 1
```

The decompressor compares the payload against `0x1e << 10`, extracts the
10-bit length field, then copies from the previous frame at the current output
position.

This command can represent 1..1024 pixels. The compressor emits multiple
commands for longer runs.

## Long Literal Run

For literal runs longer than 15 pixels, `PROCn` switches from halfword literal
pixels to packed 15-bit pixels:

```basic
PROCopbw(1+(&1f<<11)+(len%-1<<1))
FORJ%=newz%TOZ%-4STEP4
  PROCopb(b2%!J%)
NEXT
bop%=(bop%+15ANDNOT15)>>3:REM word align
```

The command header is:

```text
bit 0       = 1
bits 1..10  = length - 1
bits 11..15 = 0x1f
```

It is followed by `length` packed 15-bit pixels. After the packed pixels, the
stream is realigned to the next 16-bit boundary.

The compressor only uses this form for literal runs longer than 15 pixels.
Shorter runs are written as individual literal halfwords.

The decompressor's `decn` path reads these packed pixels through a bit pointer
and then realigns:

```basic
ADD r0,r0,#15
BIC r0,r0,#15 REM realign
MOV r0,r0,LSR #3 REM back to bytes
```

## Repeated Pixel Run

When the compressor sees a run of repeated source pixels longer than `rle%`, it
emits a command plus one following pixel halfword:

```basic
Q%=b2%!Z%
PROCopdw(1+(&1cc<<7)+(M%-2<<1)+(Q%<<16))
```

The first halfword is:

```text
bit 0       = 1
bits 1..6   = repeat_count - 2
bits 7..15  = 0x1cc
```

The second halfword is the repeated 15-bit pixel value.

The frame terminator uses the same family code and zero length field, so a
repeat command with this family is only meaningful when the length field is
non-zero:

```text
repeat_count = length_field + 2
```

Therefore this family can represent repeated-pixel runs of 3..65 pixels.

The first-frame path lowers the RLE threshold:

```basic
rle%=8:E%=both%:IFframe%=0 E%=spatial%:rle%=2
```

So repeated-pixel runs are easier to trigger in frame zero, where temporal
copying is unavailable.

## End Of Frame

The frame terminator is:

```text
word = 1 + (0x1cc << 7)
```

The compressor writes:

```basic
PROCopw(1+(&1cc<<7)):REM complete frame
```

The decompressor returns from the frame when the decoded command payload is
exactly:

```text
payload == 0x1cc << 6
```

That is the same halfword, viewed after shifting right by one bit.

## Code Family Summary

| Halfword pattern | Meaning |
| --- | --- |
| `(pixel << 1)` | one literal 15-bit pixel |
| `1 + ((len - 2) << 1) + (code << 7)`, `code < 0x120` | temporal copy, length 2..65 |
| `1 + ((len - 2) << 1) + (code << 7)`, `0x120 <= code < 0x1cc` | spatial copy, length 2..65 |
| `1 + ((len - 1) << 1) + (0x1e << 11)` | same-position previous-frame copy, length 1..1024 |
| `1 + ((len - 1) << 1) + (0x1f << 11)` plus packed pixels | long literal run, length 1..1024; compressor uses for `len > 15` |
| `1 + ((len - 2) << 1) + (0x1cc << 7)` plus pixel | repeated pixel, length 3..65 |
| `1 + (0x1cc << 7)` | end of frame |

## Resolved And Deferred Points

- Treat `0x1cd..0x1df` as decoder-tolerated aliases, not canonical stream
  values. A new encoder emits repeated-pixel runs only with family `0x1cc`.
- Pixel interpretation, threshold tables, and the `lrpc%`, `lrpf%`, and
  `rchk%` search behaviour are covered by the compressor and decompressor
  notes written after this first pass.
- The RO3.71/2003 source comparison is parked unless a concrete compatibility
  discrepancy appears.
