# Moving Lines Compressor Process

This note documents the first-pass compressor decision process for Acorn Replay
type 1, `Moving Lines`, from:

- `ARMovie_2003/Video/MovingLine/bas/BatchComp,ffb.txt`
- `ARMovie_RO371/Resources/MovingLine/BatchComp,ffb.txt`

It should be read with `notes/moving-lines-bitstream-first-pass.md`.

## High-Level Model

Moving Lines is a horizontal run matcher.

Unlike Moving Blocks, it does not divide the frame into fixed 4x4 blocks. It
scans the frame left-to-right, top-to-bottom, and at each pixel position tries
to encode a run of pixels using one of these modes:

- copy a run from the previous decoded frame;
- copy a run from an earlier line in the current decoded frame;
- copy a same-position run from the previous decoded frame;
- encode a repeated-pixel run;
- encode literal pixels.

The compressor also reconstructs the current decoded frame into `b3%` while it
encodes. That reconstructed frame is the source for spatial copies and becomes
the next frame's temporal reference.

## Frame Buffers

The main buffers are:

- `b1%`: previous/current reconstructed output frame, used as temporal source.
- `b2%`: source frame expanded by CompLib.
- `b3%`: reconstructed output frame being built.
- `b4%`: saved alternate reconstruction during retry fallback.

The core frame loop is:

```basic
PROCExpandNextSourceFrame(frame%)
PROCmatch
SWAP b1%,b3%
PROCCheckChunkFinished(b1%)
```

## Per-Frame Setup

`PROCmatch` sets:

```basic
rle%=8:E%=both%:IFframe%=0 E%=spatial%:rle%=2
```

So after frame zero, it searches both temporal and spatial candidates. On frame
zero there is no previous decoded frame, so it searches spatial candidates only
and lowers the repeated-pixel threshold.

Each retry clears the output frame and starts again:

```basic
PROCdothresholds
CALLinit%,b3%
Z%=0
skiplen%=0
bop%=outlen%
innew%=FALSE
!buc%=b1%
!buf%=b3%
```

`Z%` is the byte offset into the word-per-pixel frame buffers, so one pixel
advances `Z%` by 4.

## Main Pixel Scan

For every pixel position, the compressor first calls either `both%` or
`spatial%`:

```basic
!adS%=b2%+Z%
CALLE%
IFM%>1 THEN
  ...
ELSE
  ...
ENDIF
```

`M%` is the selected match length. If it is greater than one, the compressor
emits a copy/run mode. Otherwise it tries repeated-pixel coding with `rchk%`,
and finally falls back to literal output.

The mode decision order is:

1. temporal/spatial line match of length at least 2;
2. same-position previous-frame run, if the chosen match is the temporal centre;
3. repeated-pixel run;
4. literal accumulation.

## Temporal And Spatial Search

The search setup is:

```basic
move%=8
centre%=move%*(2*move%+1)+move%
smove%=9
```

Temporal search covers a 17x17 square around the current pixel in the previous
decoded frame:

```basic
lrpc%:
  y = current_y - move .. current_y + move
  x = current_x - move .. current_x + move
  source = b1%
  Q base = 0
```

Spatial search covers rows above the current output line:

```basic
lrpf%:
  y = current_y - smove .. current_y - 1
  x = current_x - smove .. current_x + smove
  source = b3%
  Q base = 1024
```

The emitted code tables are generated by `PROCg`:

```basic
FORJ%=-move% TO move%
 FORI%=-move% TO move%
  Q%+=1
  j%(Q%)=(J%*sx%+I%)*4
  IFQ%>centre% q%(Q%)=((Q%-1)<<7)+1 ELSE q%(Q%)=(Q%<<7)+1
 NEXT
NEXT

FORJ%=-smove% TO -1
 FORI%=-smove% TO smove%
  Q%+=1
  j%(Q%+1024)=(J%*sx%+I%)*4
  q%(Q%+1024)=(Q%+&120<<7)+1
 NEXT
NEXT
```

The temporal centre is deliberately not emitted as an ordinary temporal copy
code. If the best match is the centre, the compressor accumulates a
same-position previous-frame run and emits the `0x1e` family documented in the
bitstream note.

## Match Length Scoring

The search routine `lrp%` tries candidate line starts and extends each
candidate horizontally until it fails the current threshold table.

It stores the best match in a packed value:

```text
high 16 bits: accepted run length
low bits: "goodness" and dubious-match bookkeeping
```

After search, callers use:

```basic
MOV r9,r9,LSR #16
```

to recover the selected length.

Important constants:

```basic
dubious=2
big%=65
```

For normal temporal/spatial offsets, `big%=65` caps the useful run length,
matching the 6-bit copy-command length field. The temporal centre path uses
the `nolim` branch, because same-position runs can be emitted with the longer
`0x1e` command family.

Each candidate pixel comparison computes a squared component distance with
`FNdiff`, then compares that distance with the threshold table entry for the
source pixel being matched:

```basic
FNdiff(...)
LDRB r12,[r2,r6]
CMP r3,r12,LSL #1
```

The first pixel has a quick rejection at twice the threshold. Accepted pixels
are classed roughly as:

- good: error <= threshold / 2;
- acceptable: error <= threshold;
- dubious: error > threshold but not beyond the early rejection limits.

The code permits limited dubious matching and then requires the following pixel
to pass normally. This prevents one marginal pixel from ending a useful line
run immediately, but avoids letting a whole run drift badly.

## Temporal-Then-Spatial Ordering

`both%` searches temporal candidates first:

```basic
BL lrpc%
```

If the temporal result has not already reached `big%`, it then searches spatial
candidates:

```basic
CMP r9,#big%<<16
BLCC lrpf%
```

So a maximal temporal line run avoids spatial search. Otherwise, spatial
search can replace the current best match if it scores better.

`spatial%`, used on frame zero, only calls `lrpf%`.

## Copy Reconstruction

Once a copy match is chosen, the compressor immediately copies reconstructed
pixels into `b3%`.

For non-centre copies:

```basic
CALLp3copy%,j%(Q%)
PROCopw(q%(Q%)+(M%-2<<1))
```

`p3copy%` chooses the source buffer by `Q%`:

```basic
Q < 1024  -> b1%, previous decoded frame
Q >= 1024 -> b3%, current decoded frame
```

For centre temporal copies:

```basic
CALLp1copy%,j%(Q%)
skiplen%+=M%
```

`p1copy%` always copies from `b1%`, while `skiplen%` is later emitted as the
same-position previous-frame run.

## Repeated-Pixel Runs

If no line match of length at least two is found, the compressor calls `rchk%`.

`rchk%` compares following source pixels against the first source pixel. It
uses a derived threshold:

```basic
threshold = threshold_table[first_pixel] * 3 / 4 + 1
```

The result is capped at 65 pixels, matching the repeated-pixel command length.

The compressor emits a repeated-pixel run only when:

```basic
M% > rle%
```

where `rle%` is normally 8, but is 2 on frame zero or after the emergency
bitrate "Cut" path.

## Literal Accumulation

If no copy or repeated-pixel run is chosen, the current source pixel is copied
literally into `b3%` and a literal span is opened or extended:

```basic
IFNOTinnew% newz%=Z%:innew%=TRUE
b3%!Z%=b2%!Z%
Z%+=4
```

When a later non-literal mode is chosen, `PROCn` flushes the literal span:

- up to 15 pixels: one literal halfword per pixel;
- more than 15 pixels: one long literal-run command plus packed 15-bit pixels.

## Threshold Tables

`numtables%=36`, and the allocated cache covers table slots `0..35`.
The source also contains a quality-mode clamp that can set `qual%` to
`numtables%`, while the retry loop later clamps values above `numtables%-1`
back down. Treat the practical table range as `0..35` unless a later pass finds
that the `36` case is reachable safely.

Each threshold table is 32768 bytes: one byte for every possible 15-bit pixel
value.

```basic
DIM tar%(numtables%-1)
DIM tabs% 32768*numtables%-1
```

Tables are generated lazily in `PROCdothresholds`. `fine%` and `coarse%` both
point to the current table:

```basic
!fine=fine%
!coarse=fine%
```

So although the assembler names the temporal table "coarse" and the spatial
table "fine", this version uses the same threshold table for both searches.

For YUV sources, the threshold generator treats U and V as signed 5-bit
differences around zero:

```basic
v%=b%:IFv%>15 v%-=32
u%=g%:IFu%>15 u%-=32
y%=u%^2+v%^2
```

For RGB sources, it treats all three 5-bit components as unsigned component
differences:

```basic
y%=b%^2+g%^2
```

The actual match distance from `FNdiff` mirrors this distinction. For YUV it
wraps the chroma components around the signed centre; for RGB it uses straight
absolute component differences.

Higher `qual%` generally produces looser thresholds, more approximate line
matches, and smaller output.

## Frame Retry Control

Moving Lines uses the same broad retry strategy as Moving Blocks:

- encode the whole frame at the current `qual%`;
- if the frame is too small, lower `qual%` for stricter matching and larger
  output;
- if the frame is too large, raise `qual%` for looser matching and smaller
  output;
- save a "small but OK" attempt so it can fall back if later retries overshoot.

The normal target window is:

```basic
IFfps<25 THEN
  uplim%=downlim%-300
  downlim%=downlim%+300
ELSE
  uplim%=downlim%-150
  downlim%=downlim%+150
ENDIF
```

If `qual%` rises three levels above the frame's initial value, the compressor
enters a cut path:

```basic
IFqual%-iq%=3 THEN
  downlim2%=downlim%+x%
  rle%=2
ENDIF
```

This relaxes the byte target and makes repeated-pixel runs easier to use.

## RO3.71 Versus 2003

The 2003 `BatchComp` is easier to read because it has more comments and
spacing. Its version string is unchanged from RO3.71:

```basic
MovingLines Batch Compressor v 0.24 08th April 1994
```

The source-level compressor structure appears the same at this pass: same
CompLib entry points, same code families, same search variables, same threshold
generation, and same retry logic.

There are some assembly return-sequence differences in the detokenised text,
for example RO3.71 uses `LDMFD ...,{pc}^` in places where the 2003 source uses
an explicit `CMP r0,r0` to clear V followed by `LDMFD ...,{pc}`. These need a
separate generated-code comparison before being classified as semantic or
toolchain-related.

## C Implementation Implications

Moving Lines should not share the Moving Blocks block-search core. It can share
the outer Replay tooling:

- frame input and colour conversion;
- reconstructed-frame buffers;
- retry/rate-control framework;
- Replay container writing;
- tracing and verifier infrastructure.

The codec core should be separate and line-run based:

- scan one pixel position at a time;
- search horizontal runs in previous/current reconstructed frames;
- emit command halfwords, not an LSB-first bitstream;
- keep same-position previous-frame runs as a distinct mode;
- keep literal spans open until a copy/RLE mode forces a flush.
