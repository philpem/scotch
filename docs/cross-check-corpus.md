# Cross-Check Corpus

## Bundled Replay Movies

The repository contains Replay movies under:

```text
RiscOS_2003/RiscOS/UnU/Demos/Videos/ARMovies
```

The inspected video files all identify their video format as type 7, Moving
Blocks. They are suitable future fixtures for `codec_movingblocks`, but they do
not provide an independent type 19, Super Moving Blocks corpus.

## Current Type 19, Super Moving Blocks Coverage

The bundled compiled decoder is now exercised under Unicorn for focused
stationary, temporal, spatial, split, and lossy C-encoded payloads. This proves
those payloads decode identically in the portable verifier and Acorn
implementation. It does not prove that the C encoder chooses the same modes as
Acorn's compressor; that is a documented comparison target rather than a
compatibility requirement.

The independent encoder-policy corpus was supplied as `LionFish19,ae7`, encoded
with the original type 19, Super Moving Blocks compressor. The tooling now
parses its AE7 catalogue and has decoded and split all 25 frames in chunk 0 by
following the source pointers returned by compiled Decomp19.

The portable verifier and compiled Acorn decoder agree byte-for-byte on those
25 frames. The first two frames are checked in: frame 0 has no temporal
dependency, while frame 1 uses frame 0's reconstructed output. Their payload,
reference, output, source-movie, and compiled-decoder hashes are recorded in
`corpus/format19/manifest.tsv` and the corpus runs under CTest.

Further decision-policy work can use:

- payloads generated by the original Decomp19 `BatchComp` compressor;
- a small RISC OS harness that runs the original compressor on synthetic
  frames, followed by either decompressor path.

The first requested original-compressor sample is `LionFish2,ae7`, recompressed
as type 19, Super Moving Blocks at fixed quality level 7 with dithering
disabled. Exact settings and preservation requirements are in
[acorn-comparison-workflow.md](acorn-comparison-workflow.md).

The source sample is `LionFish19,ae7` (3,554,760 bytes,
SHA-256 `e4a6539b19a105e80e3171a4753870b184edafded0ee874bf2f470231b661684`).
It is type 19, Super Moving Blocks, 160x128, 12.5 frames/s, with 25 frames per
chunk and 15 catalogue entries. Chunk 0 declares 181,886 video bytes; 25
decoder returns consume 181,885 bytes, leaving one byte of chunk padding.

Any imported fixture should record:

- source path or generation command;
- width and height;
- frame number and previous-frame dependency;
- exact compressed payload bytes;
- decoded `6Y5UV` bytes from the original decompressor;
- checksum of each artifact.

Do not use payloads produced by the portable writer as the sole compatibility
oracle. They are appropriate unit-test fixtures, but independent validation
must ultimately pass through the Acorn decompressor.
